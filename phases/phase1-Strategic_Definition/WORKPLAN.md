# Phase 1 - Strategic Definition

## Purpose

This phase defines what ADF is at the highest level before any architecture, workflow, or implementation work begins.

## Core Question

What are we building, for whom, why now, and what must be true for it to be worth building?

## Scope

- vision
- target users and stakeholders
- problem statement
- value proposition
- success criteria
- boundaries and non-goals
- high-level requirements

## Out Of Scope

- architecture
- detailed workflows
- technical design
- implementation planning

## Main Outputs

- vision statement
- problem and user definition
- high-level goals
- prioritized requirements list
- non-goals
- initial success metrics

## Steps

1. Define the problem first: what pain exists, for whom, and why it matters now.
2. Define the ambition: what ADF should become if it succeeds in the strongest possible version.
3. Define the user and buyer clearly: who gets value, who decides, and who operates it.
4. Define the core promise: the single outcome ADF must reliably deliver better than alternatives.
5. Define success in measurable terms: business outcomes, user outcomes, and 3-5 north-star metrics.
6. Define the boundaries: explicit non-goals, exclusions, and what ADF will not try to be.
7. Define the key use cases: the small set of high-value jobs ADF must support end to end.
8. Define the operating model: main actors, roles, and how work and value should flow through the system.
9. Define the product principles: a few hard rules that future decisions must follow.
10. Define the risks and constraints: legal, technical, organizational, trust, speed, cost, and complexity limits.
11. Define the release thesis: what must be true for v1 to matter, and what can wait until later phases.
12. Turn all of that into high-level requirements: required capabilities, required qualities, required integrations, and required governance.
13. Rank the requirements: must-have, should-have, nice-to-have, and explicitly cut the rest.
14. Freeze the top layer before going deeper: do not start architecture, workflows, or implementation until vision, scope, and high-level requirements are approved.

## Exit Criterion

A new person can read the top-level phase outputs and clearly answer what ADF is, why it exists, what it must do, and what it will not do.

## Rule For This Phase

If a discussion starts with how should we build it, it belongs to the next phase.
